Introduction: The Promise of Lakebase

For years, data architecture has been split into two camps: OLAP (Online Analytical Processing) for data warehousing and analytics, and OLTP (Online Transaction Processing) for operational databases. Organizations have built complex ETL pipelines to synchronize data between these worlds. But what if you could eliminate that separation?

Databricks Lakebase changes the game. It's a managed PostgreSQL 16 OLTP database that runs natively within Databricks, allowing you to perform transactional operations without leaving your lakehouse. This opens up an entirely new category of applications: real-time systems that read analytical data from Delta Lake and write transactional updates to a single platform.

In this tutorial, we'll build the PL Live Scout Hub—a production-grade Streamlit application that scouts use to review player analytics, submit real-time reports, and collaborate on recruitment decisions. The app demonstrates how Lakebase bridges the OLAP-OLTP divide, enabling instant visibility across both analytical and transactional layers without complex integration work.

What You'll Learn

By the end of this tutorial, you'll understand how to architect and deploy a real-time OLTP+OLAP application, including provisioning managed PostgreSQL, syncing Delta tables, building a Streamlit frontend, and executing cross-layer queries that unite analytics with operations.

The Use Case: Premier League Scouting

Imagine you're building software for Premier League scouts. They need:

  • Analytical access: Player statistics, performance trends, historical match events (data that changes slowly and is queried heavily)
  • Transactional capability: Real-time scout reports, player watchlists, transfer recommendations (data that changes frequently and requires ACID guarantees)
  • Real-time correlation: The ability to ask, "Which unscouted players have the best expected goals metrics?" or "Do scout ratings correlate with actual performance?"

The traditional approach would involve:

  1. Maintain player stats and match events in a data warehouse (Databricks Delta Lake)
  2. Build an operational database elsewhere (AWS RDS, Azure SQL, Postgres) for scout reports and watchlists
  3. Create ETL pipelines to synchronize data between systems (introducing latency and complexity)
  4. Build complex application logic to correlate data across multiple systems

The Lakebase approach:

  1. Keep analytical data in Delta Lake (unchanged)
  2. Provision Lakebase as your OLTP database within Databricks
  3. Sync Delta tables to Lakebase schemas (automatic, low-latency)
  4. Create OLTP tables in Lakebase for scout operations
  5. Build your app to query across both layers in a single SQL statement

No ETL pipelines. No data synchronization delays. No separate operational database. Just one unified platform.

Architecture Overview

The PL Live Scout Hub is built on a three-layer architecture:

Layer 1: Delta Lake + Unity Catalog (OLAP)

This is your analytical foundation. We'll load:

  • players table: 210 Premier League players with static attributes
  • season_stats table: Current season performance metrics (goals, assists, expected goals, etc.)
  • match_events table: ~2000 match event records (passes, shots, tackles, defensive actions)

These tables live in Delta Lake with Unity Catalog governance. They're immutable from the app's perspective (scouts don't modify stats directly).

Layer 2: Lakebase Synced Tables (OLAP in OLTP)

Lakebase automatically syncs specified Delta tables into its own schemas. When you create a synced table, Lakebase:

  • Creates a read-only replica of the Delta table in Postgres
  • Maintains low-latency synchronization in the background
  • Places the replica in a dedicated pl_data schema

Scouts query these tables to see player analytics. From the app's perspective, it's just another schema in the same Postgres database.

Layer 3: Lakebase OLTP Tables (Transactional)

In a separate scout schema, we create native Postgres tables for scout operations:

  • scout_reports: Player evaluations submitted by scouts (player_id, rating, notes, timestamp)
  • player_watchlist: Tracked players of interest
  • transfer_recommendations: Formal recommendations to management

These tables have full ACID semantics, support constraints, and accept INSERT/UPDATE/DELETE operations from your application.

Frontend: Databricks Apps (Streamlit)

The user-facing layer. We'll build a Streamlit application with:

  • Player Dashboard: Browse all players with sorting and filtering
  • Player Profile: Detailed stats with a form to submit scout reports
  • Scout Board: View all scout reports and watchlist
  • Analytics Hub: Cross-layer queries combining synced stats with scout operations
  • Settings: Configuration and admin features

The app connects to Lakebase via standard PostgreSQL (psycopg2), using Databricks service principal authentication.

Step 3: Provision Lakebase

Now we'll provision a managed PostgreSQL 16 database within Databricks. This is where the magic happens.

Prerequisites

You'll need:

  • The databricks-sdk for Python (version >= 0.61.0)
  • Databricks workspace admin permissions
  • A Databricks personal access token or OAuth token

Create the Provisioned Instance

In a Databricks notebook, run:

from databricks.sdk import WorkspaceClient

w = WorkspaceClient()

# Create a Lakebase provisioned instance
instance = w.provisioned_database_endpoints.create(
    name="pl-scout-lakebase",
    cluster_size="SMALL"  # 2 CPUs, 8GB RAM. Can scale up later.
)

print(f"Instance ID: {instance.id}")
print(f"JDBC URL: {instance.jdbc_url}")

# Store the instance ID for later use
instance_id = instance.id

This creates a PostgreSQL 16 server that you can connect to via port 5432. The instance will be ready in 2-5 minutes.

Test the Connection

Once the instance is ready, generate credentials and test the connection:

import psycopg2

# Generate temporary database credentials
creds = w.database.generate_database_credential(instance_id=instance_id)

# Extract connection details
host = instance.host
port = 5432
user = creds.username
password = creds.password
database = "postgres"

# Test the connection
try:
    conn = psycopg2.connect(
        host=host,
        port=port,
        user=user,
        password=password,
        database=database,
        sslmode="require"
    )
    cursor = conn.cursor()
    cursor.execute("SELECT version();")
    print("Connected! PostgreSQL version:")
    print(cursor.fetchone())
    cursor.close()
    conn.close()
except Exception as e:
    print(f"Connection failed: {e}")
Important

Always use sslmode=require when connecting to Lakebase from outside the workspace. Do not disable SSL, even for testing.

Step 4: Set Up Synced Tables

Synced tables are the bridge between Delta Lake and Lakebase OLTP. They create read-only replicas of your analytical tables in the PostgreSQL database.

Create Synced Tables via REST API

Use the Databricks REST API to create synced tables:

import requests
import json

# Get your Databricks workspace URL and token
workspace_url = "https://your-workspace.databricks.com"
token = "dapi..."  # Your PAT token

# Function to create a synced table
def create_synced_table(source_table, target_schema):
    url = f"{workspace_url}/api/2.0/database/synced_tables"

    headers = {
        "Authorization": f"Bearer {token}",
        "Content-Type": "application/json"
    }

    body = {
        "provisioned_database_instance_id": instance_id,
        "schema_name": target_schema,
        "table_name": source_table.split(".")[-1],
        "source_table_full_name": source_table
    }

    response = requests.post(url, headers=headers, json=body)
    return response.json()

# Create synced tables in pl_data schema
result1 = create_synced_table("main.pl_data.players", "pl_data")
result2 = create_synced_table("main.pl_data.season_stats", "pl_data")
result3 = create_synced_table("main.pl_data.match_events", "pl_data")

print("Synced tables created!")
print(f"Players: {result1}")
print(f"Season stats: {result2}")
print(f"Match events: {result3}")

After running this, the pl_data schema in your Lakebase instance will contain read-only replicas of the three tables. Lakebase automatically keeps them synchronized.

Tip

Synced tables are ideal for analytical data that changes infrequently. For high-frequency updates, consider replicating with a shorter sync interval or implementing custom ETL logic.

Step 5: Create OLTP Tables

Now we'll create native PostgreSQL tables for scout operations. These will have full transactional semantics.

Connect to Lakebase and Create Tables

In a Databricks notebook:

import psycopg2

# Generate fresh credentials
creds = w.database.generate_database_credential(instance_id=instance_id)

# Connect to the database
conn = psycopg2.connect(
    host=instance.host,
    port=5432,
    user=creds.username,
    password=creds.password,
    database="postgres",
    sslmode="require"
)

cursor = conn.cursor()

# Create the scout schema
cursor.execute("CREATE SCHEMA IF NOT EXISTS scout;")

# Create scout_reports table
cursor.execute("""
    CREATE TABLE IF NOT EXISTS scout.scout_reports (
        report_id SERIAL PRIMARY KEY,
        player_id TEXT NOT NULL,
        scout_name TEXT NOT NULL,
        rating DECIMAL(3,1) CHECK (rating >= 0 AND rating <= 10),
        physical_score DECIMAL(3,1),
        technical_score DECIMAL(3,1),
        tactical_score DECIMAL(3,1),
        notes TEXT,
        video_link TEXT,
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
        updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
        FOREIGN KEY (player_id) REFERENCES pl_data.players(player_id)
    );
""")

# Create player_watchlist table
cursor.execute("""
    CREATE TABLE IF NOT EXISTS scout.player_watchlist (
        watchlist_id SERIAL PRIMARY KEY,
        player_id TEXT NOT NULL,
        priority TEXT CHECK (priority IN ('High', 'Medium', 'Low')),
        reason TEXT,
        added_by TEXT,
        added_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
        UNIQUE(player_id),
        FOREIGN KEY (player_id) REFERENCES pl_data.players(player_id)
    );
""")

# Create transfer_recommendations table
cursor.execute("""
    CREATE TABLE IF NOT EXISTS scout.transfer_recommendations (
        recommendation_id SERIAL PRIMARY KEY,
        player_id TEXT NOT NULL,
        recommendation_type TEXT CHECK (recommendation_type IN ('Buy', 'Sell', 'Loan')),
        rationale TEXT,
        estimated_fee DECIMAL(12,2),
        confidence_score DECIMAL(3,1),
        recommended_by TEXT,
        status TEXT DEFAULT 'Open',
        created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
        FOREIGN KEY (player_id) REFERENCES pl_data.players(player_id)
    );
""")

conn.commit()
print("OLTP tables created successfully!")

cursor.close()
conn.close()
Important Type Alignment

Notice that player_id is TEXT in all tables. This is critical—the synced tables also have player_id as TEXT, so we need exact type matching for JOINs to work correctly.

db.py – Database Connection Module

import psycopg2
from psycopg2 import sql
import pandas as pd
from databricks.sdk import WorkspaceClient

class LakebaseConnection:
    def __init__(self, instance_id):
        self.w = WorkspaceClient()
        self.instance_id = instance_id
        self.conn = None

    def connect(self):
        """Establish connection using Databricks credentials"""
        # Generate temporary credentials
        creds = self.w.database.generate_database_credential(
            instance_id=self.instance_id
        )

        # Get instance details
        instance = self.w.provisioned_database_endpoints.get(self.instance_id)

        # Connect
        self.conn = psycopg2.connect(
            host=instance.host,
            port=5432,
            user=creds.username,
            password=creds.password,
            database="postgres",
            sslmode="require"
        )

    def execute_query(self, query, params=None):
        """Execute a query and return results as DataFrame"""
        if not self.conn:
            self.connect()

        cursor = self.conn.cursor()
        try:
            cursor.execute(query, params)
            columns = [desc[0] for desc in cursor.description]
            rows = cursor.fetchall()
            df = pd.DataFrame(rows, columns=columns)
            self.conn.commit()
            return df
        except Exception as e:
            self.conn.rollback()
            raise e
        finally:
            cursor.close()

    def insert_scout_report(self, player_id, scout_name, rating,
                           physical_score, technical_score, tactical_score, notes):
        """Insert a scout report"""
        query = """
            INSERT INTO scout.scout_reports
            (player_id, scout_name, rating, physical_score, technical_score,
             tactical_score, notes)
            VALUES (%s, %s, %s, %s, %s, %s, %s)
        """
        params = (player_id, scout_name, rating, physical_score,
                 technical_score, tactical_score, notes)

        cursor = self.conn.cursor()
        try:
            cursor.execute(query, params)
            self.conn.commit()
            return True
        except Exception as e:
            self.conn.rollback()
            raise e
        finally:
            cursor.close()

app.py – Main Application

import streamlit as st
from db import LakebaseConnection
import os

# Configure page
st.set_page_config(
    page_title="PL Live Scout Hub",
    layout="wide",
    initial_sidebar_state="expanded"
)

# Initialize connection
@st.cache_resource
def get_db():
    instance_id = os.getenv("LAKEBASE_INSTANCE_ID")
    if not instance_id:
        st.error("LAKEBASE_INSTANCE_ID environment variable not set")
        st.stop()
    return LakebaseConnection(instance_id)

db = get_db()
db.connect()

# Header
st.markdown("# PL Live Scout Hub")
st.markdown("*Powered by Databricks Lakebase*")

# Sidebar navigation
page = st.sidebar.radio(
    "Navigate",
    ["Player Dashboard", "Player Profile", "Scout Board",
     "Analytics Hub", "Settings"]
)

if page == "Player Dashboard":
    st.header("Player Dashboard")

    # Fetch all players
    players_df = db.execute_query(
        "SELECT player_id, name, position, team, nationality FROM pl_data.players"
    )

    # Filters
    col1, col2, col3 = st.columns(3)

    with col1:
        position_filter = st.multiselect(
            "Position",
            players_df['position'].unique()
        )

    with col2:
        team_filter = st.multiselect(
            "Team",
            players_df['team'].unique()
        )

    # Apply filters
    filtered = players_df
    if position_filter:
        filtered = filtered[filtered['position'].isin(position_filter)]
    if team_filter:
        filtered = filtered[filtered['team'].isin(team_filter)]

    st.dataframe(filtered, use_container_width=True)

elif page == "Analytics Hub":
    st.header("Analytics Hub")

    # Cross-layer query: scouted vs unscouted players with stats
    query = """
        SELECT
            p.name,
            p.position,
            p.team,
            ss.goals,
            ss.assists,
            ss.xg,
            COUNT(sr.report_id) as scout_reports,
            ROUND(AVG(sr.rating), 1) as avg_rating
        FROM pl_data.players p
        LEFT JOIN pl_data.season_stats ss ON p.player_id = ss.player_id
        LEFT JOIN scout.scout_reports sr ON p.player_id = sr.player_id
        GROUP BY p.player_id, p.name, p.position, p.team,
                 ss.goals, ss.assists, ss.xg
        ORDER BY COUNT(sr.report_id) DESC
    """

    analytics_df = db.execute_query(query)
    st.dataframe(analytics_df, use_container_width=True)

The Magic: Cross-Layer Queries

This is where Lakebase demonstrates its power. In a single SQL query, we can join analytical data (from synced tables) with transactional data (from OLTP tables). Let's explore some examples:

Example 1: Scouted vs Unscouted Players

SELECT
    p.name,
    p.position,
    ss.xg,
    ss.goals,
    COUNT(sr.report_id) as total_scout_reports,
    ROUND(AVG(sr.rating), 1) as avg_scout_rating,
    CASE
        WHEN COUNT(sr.report_id) = 0 THEN 'Unscouted'
        WHEN COUNT(sr.report_id) < 3 THEN 'Lightly Scouted'
        ELSE 'Well Scouted'
    END as scout_status
FROM pl_data.players p
LEFT JOIN pl_data.season_stats ss ON p.player_id = ss.player_id
LEFT JOIN scout.scout_reports sr ON p.player_id = sr.player_id
GROUP BY p.player_id, p.name, p.position, ss.xg, ss.goals
ORDER BY total_scout_reports DESC, ss.xg DESC;

This query instantly shows which high-performing players (high xG) haven't received scouting attention yet. In a traditional architecture, you'd need to:

  1. Query the data warehouse for stats
  2. Query the operational database for scout reports
  3. Correlate in application code

With Lakebase, it's a single query executed in PostgreSQL.

Example 2: Scout Rating vs Performance Correlation

SELECT
    sr.scout_name,
    COUNT(*) as players_rated,
    ROUND(AVG(sr.rating), 2) as avg_rating,
    ROUND(AVG(ss.xg), 2) as avg_target_xg,
    ROUND(CORR(sr.rating, ss.xg), 3) as correlation_to_xg,
    ROUND(CORR(sr.rating, ss.goals), 3) as correlation_to_goals
FROM scout.scout_reports sr
JOIN pl_data.season_stats ss ON sr.player_id = ss.player_id
GROUP BY sr.scout_name
ORDER BY players_rated DESC;

This tells you whether a scout's ratings align with actual performance metrics. Are high-rated scouts identifying players who truly perform?

Example 3: Transfer Targets with Advanced Analytics

SELECT
    p.name,
    p.team,
    ss.xg,
    ss.goals,
    tr.estimated_fee,
    tr.confidence_score,
    CASE
        WHEN ss.xg / NULLIF(ss.goals, 0) > 1.2 THEN 'Underperforming'
        WHEN ss.xg / NULLIF(ss.goals, 0) < 0.8 THEN 'Overperforming'
        ELSE 'Aligned'
    END as performance_status,
    me.success_rate
FROM scout.transfer_recommendations tr
JOIN pl_data.players p ON tr.player_id = p.player_id
JOIN pl_data.season_stats ss ON p.player_id = ss.player_id
LEFT JOIN LATERAL (
    SELECT
        ROUND(100.0 * SUM(CASE WHEN success THEN 1 ELSE 0 END) / COUNT(*), 1) as success_rate
    FROM pl_data.match_events
    WHERE player_id = p.player_id
) me ON true
WHERE tr.status = 'Open'
ORDER BY tr.confidence_score DESC;

This combines transfer recommendations with performance analysis and match event success rates to identify the best transfer targets.

Key Learnings and Gotchas

1. Synced Tables Go to Their Own Schema

When you create a synced table from main.pl_data.players, Lakebase doesn't place it in a default schema. You must specify the target schema (e.g., pl_data). The replica won't appear in the public schema.

2. Type Alignment Matters

If your Delta table has player_id as STRING, the synced table will have player_id as TEXT. Your OLTP tables must match exactly. A VARCHAR mismatch can cause JOIN failures.

3. App Service Principal Needs Explicit Grants

By default, a Databricks App service principal has limited access. You must explicitly GRANT permissions on schemas:

GRANT SELECT ON SCHEMA pl_data TO `service-principal-name`;
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA scout TO `service-principal-name`;

4. databricks-sdk >= 0.61.0 Required

The database.generate_database_credential() method was added in SDK version 0.61.0. Older versions won't work. Always pin your requirements.txt to a recent version.

5. Authentication via generate_database_credential()

Do not use personal access tokens (PATs) for database connections. Instead, use the Databricks SDK's generate_database_credential() method, which issues temporary, scoped credentials. This is more secure and works seamlessly in Databricks Apps.

6. SSL Mode is Required

Always use sslmode=require when connecting from outside the workspace. Databricks enforces encrypted connections to Lakebase instances.

7. Synced Tables Are Read-Only

You cannot INSERT, UPDATE, or DELETE data in synced tables. They're replicas meant for analysis. If you need to write to them, use the original Delta tables and let Lakebase sync the changes.

8. Connection Pooling in Multi-User Apps

For production Streamlit apps, consider implementing connection pooling to avoid exhausting database connections. Streamlit reruns the entire script on user interaction, so create connections carefully:

@st.cache_resource
def get_db_connection():
    """Cached connection to avoid multiple connections"""
    instance_id = os.getenv("LAKEBASE_INSTANCE_ID")
    db = LakebaseConnection(instance_id)
    db.connect()
    return db
